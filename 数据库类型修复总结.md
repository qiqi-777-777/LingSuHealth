# 数据库外键约束修复总结

## 问题描述
创建 `user_daily_tasks` 表时报错：
```
Foreign key constraint 'fk_long_user' is incorrectly formed
```

## 根本原因
- 数据库中 `users.id` 字段类型为 `INT`
- Java 实体类 `User.java` 使用 `Long` 类型
- 新表 `user_daily_tasks.user_id` 使用 `BIGINT` 类型
- 类型不匹配导致无法创建外键约束

## 解决方案

### 1. 统一数据类型
将所有相关字段统一为 `BIGINT` / `Long`：
- 数据库：`users.id` 从 `INT` 改为 `BIGINT`
- 数据库：所有 `user_id` 字段统一为 `BIGINT`
- Java：所有实体类使用 `Long` 类型

### 2. 修改的文件

#### Java 实体类
- `src/main/java/com/lingsu/health/entity/DailyTask.java`
  - `userId` 字段从 `Integer` 改回 `Long`

#### SQL 脚本
- `scripts/create_daily_tasks_table.sql`
  - 添加外键约束：`CONSTRAINT fk_daily_task_user FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE`

- `scripts/migrate_users_id_to_bigint.sql` (新建)
  - 将 `users.id` 从 `INT` 改为 `BIGINT`
  - 更新所有相关表的 `user_id` 字段类型
  - 重新创建外键约束

#### 文档
- `scripts/README_database_migration.md` (新建)
  - 详细的迁移步骤说明
  - 备份和回滚方案

- `scripts/test_migration.sql` (新建)
  - 验证迁移结果的测试脚本

## 技术细节

### 为什么选择 BIGINT？
1. **未来扩展性**：BIGINT 可以存储更大的数值范围（-2^63 到 2^63-1）
2. **Java 兼容性**：Java 的 `Long` 类型对应数据库的 `BIGINT`
3. **行业标准**：现代应用通常使用 BIGINT 作为主键类型
4. **安全转换**：INT 到 BIGINT 的转换不会丢失数据

### 外键约束的好处
1. **数据完整性**：防止插入不存在的 user_id
2. **级联删除**：删除用户时自动删除相关任务
3. **性能优化**：数据库可以优化查询计划
